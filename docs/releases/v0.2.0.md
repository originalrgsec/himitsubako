# Release Manifest — v0.2.0

**Date:** 2026-04-11
**Tag:** `v0.2.0` (commit `6a9f519`)
**Theme:** Multi-backend track + per-credential routing

## Overview

Sprint 2. Six stories, 13 points, all delivered in a single working day. v0.2.0
turns himitsubako from "a SOPS wrapper" into "a multi-backend credential
abstraction with per-credential routing" without breaking any v0.1.x
configuration. Backward-compatibility tests in the v0.1.0 suite all pass
unchanged.

This release also folds in v0.1.1 (HMB-S017 hardening), which shipped earlier
the same day under its own tag.

This is **not on PyPI**. The public PyPI release target remains v0.3.0.

## Code Changes

### New modules

- `src/himitsubako/backends/env.py` — `EnvBackend` (read-only first-class env
  var backend with prefix filtering)
- `src/himitsubako/backends/keychain.py` — `KeychainBackend` (keyring wrapper,
  MRO-based deny-list, `list_keys()` raises by design)
- `src/himitsubako/backends/bitwarden.py` — `BitwardenBackend` (`bw` subprocess,
  strict-mode `BW_SESSION` default, optional pinned-bin and unlock_command modes)
- `src/himitsubako/router.py` — `BackendRouter` (the universal dispatcher)
- `src/himitsubako/direnv.py` — `generate_envrc`, `update_envrc`
- `src/himitsubako/pydantic.py` — `HimitsubakoSettingsSource`

### Modified modules

- `src/himitsubako/api.py` — `_resolve_backend` now returns a `BackendRouter`
  instead of branching on `default_backend`. The internal `_EnvFallbackBackend`
  shim was removed; the no-config fallback returns the real `EnvBackend()`.
- `src/himitsubako/cli/secrets.py` — same router refactor; new `hmb direnv-export`
  command; `hmb set` calls `update_envrc()` best-effort after a successful sops
  write; `hmb list` handles `BackendError` from `list_keys` (keychain) gracefully.
- `src/himitsubako/cli/init.py` — `_build_envrc` now wraps `direnv.generate_envrc`
  rather than building the .envrc inline.
- `src/himitsubako/cli/__init__.py` — registers the new `hmb direnv-export` command.
- `src/himitsubako/config.py` — added `CredentialRoute` model and
  `HimitsubakoConfig.credentials: dict[str, CredentialRoute]` field; added
  `BitwardenConfig.bin` and `BitwardenConfig.unlock_command`.

### pyproject.toml

- `version` 0.1.1 → 0.2.0
- `[project.optional-dependencies]` extras tightened to license-checked version
  ranges (`keyring>=25.0,<26.0`, `pydantic-settings>=2.13,<3.0`); `[all]` extra
  added; comment block added warning against the `bitwarden-sdk` landmine.

## Code Review Findings

`python-reviewer` was run on the cumulative `v0.1.1..HEAD` diff. Six findings,
all addressed before tag:

| # | Severity | File | Issue | Fix |
|---|---|---|---|---|
| 1 | CRITICAL | `bitwarden.py` | BW_SESSION leak via `_raise_friendly` stderr passthrough | `_redact_tokens` strips base64-like substrings (40+ chars of `[A-Za-z0-9+/=]`) before interpolating |
| 2 | CRITICAL | `bitwarden.py` | BW_PASSWORD env var design risk | `_run_bw` drops sensitive `extra_env` keys in a `finally` block (defense in depth) |
| 3 | HIGH | `keychain.py` | Dead-code outer ImportError catch | Removed; test updated to use `BackendError` side_effect |
| 4 | HIGH | `keychain.py` | Deny-list bypassable via subclassing | Switched from leaf class name to MRO walk; new regression test verifies `SafeWrapper(PlaintextKeyring)` rejected |
| 5 | HIGH | `direnv.py` | Duplicate-marker corruption silently drops user lines | `update_envrc` refuses to operate on `.envrc` with multiple markers; new regression test |
| 6 | HIGH | `direnv.py` | `secrets_file` shell injection via eval line | `shlex.quote` applied in both `generate_envrc` and `_managed_block`; two new regression tests |

The reviewer also flagged a HIGH on `HimitsubakoSettingsSource.get_field_value`
signature, claiming the parameter order was reversed. Verified against
`pydantic_settings/sources/base.py:132` — actual contract is
`(self, field: FieldInfo, field_name: str)`, which matches our implementation.
False positive; the docstring was clarified to cite the source location.

## Security

- **T-005 mitigated** (HMB-S009): `bw` binary path can be pinned via
  `HIMITSUBAKO_BW_BIN` or `bitwarden.bin` config to prevent PATH hijack.
- **T-007 partially mitigated** (HMB-S009): `BW_SESSION` is never logged or
  interpolated into error strings; `bw` stderr is sanitized to redact base64
  token-like substrings. The OS-level visibility of env vars to same-user
  processes remains an accepted limitation of the env-var session model.
- **T-008 mitigated** (HMB-S009): 30-second subprocess timeout on all `bw`
  calls, matching the SOPS pattern from v0.1.1.
- **T-020 mitigated** (HMB-S008): Keychain access delegates to the OS via the
  keyring library; first access from a new binary triggers a Touch ID /
  password prompt on macOS.
- **T-022 mitigated via M-014** (HMB-S009): Documentation guidance on safe
  `unlock_command` choices; the library does not log unlock_command output.
- **T-023 mitigated via M-015** (HMB-S008): Insecure-backend deny-list at
  first call, with MRO-based subclass detection.

## Known Limitations (carried forward)

- **No real-binary integration tests.** All Sprint 2 backends were unit-tested
  with mocked subprocesses and a mocked `keyring` module. A fresh
  `pip install himitsubako[all]` against a real Bitwarden vault has not been
  validated end-to-end. HMB-S013 in v0.3.0 covers this.
- **No CI pipeline.** Tests run locally only. HMB-S014 in v0.3.0 covers this.
- **No mkdocs documentation site.** README is the only user-facing doc.
  HMB-S015 in v0.3.0 covers this.
- **T-013 (age key permission drift) carried forward.** No `hmb status`
  command yet to detect drift; v0.2.0 review item.
- **`hmb delete`, `hmb rotate`, `hmb status` commands** still not implemented.
  Carried into v0.3.0 if real usage surfaces a need.
